Memory Chiplet

Miden VM supports linear read-write random access memory. This memory is word-addressable, meaning, four values are located at each address, and we can read and write values to/from memory in batches of four. Each value is a field element in a 646464-bit prime field with modulus 264−232+12^{64} - 2^{32} + 1264−232+1. Memory address can be any field element.

In this note we describe the rational for selecting the above design and describe AIR constraints needed to support it.

The design makes extensive use of 161616-bit range checks. An efficient way of implementing such range checks is described here.

Alternative designs​

The simplest (and most efficient) alternative to the above design is contiguous write-once memory. To support such memory, we need to allocate just two trace columns as illustrated below.

In the above, addr column holds memory address, and value column holds the field element representing the value stored at this address. Notice that some rows in this table are duplicated. This is because we need one row per memory access (either read or write operation). In the example above, value bbb was first stored at memory address 111, and then read from this address.

The AIR constraints for this design are very simple. First, we need to ensure that values in the addr column either remain the same or are incremented by 111 as we move from one row to the next. This can be achieved with the following constraint:

(a′−a)⋅(a′−a−1)=0(a' - a) \cdot (a' - a - 1) = 0(a′−a)⋅(a′−a−1)=0

where aaa is the value in addr column in the current row, and a′a'a′ is the value in this column in the next row.

Second, we need to make sure that if the value in the addr column didn't change, the value in the value column also remained the same (i.e., a value stored in a given address can only be set once). This can be achieved with the following constraint:

(v′−v)⋅(a′−a−1)=0(v' - v) \cdot (a' - a - 1) = 0(v′−v)⋅(a′−a−1)=0

where vvv is the value in value column at the current row, and v′v'v′ is the value in this column in the next row.

In addition to the above constraints we would also need to impose constraints needed for permutation checks, but we omit these constraints here because they are needed for all designs described in this note.

As mentioned above, this approach is very efficient: each memory access requires just 222 trace cells.

Read-write memory​

Write-once memory is tricky to work with, and many developers may need to climb a steep learning curve before they become comfortable working in this model. Thus, ideally, we'd want to support read-write memory. To do this, we need to introduce additional columns as illustrated below.

In the above, we added clk column, which keeps track of the clock cycle at which memory access happened. We also need to differentiate between memory reads and writes. To do this, we now use two columns to keep track of the value: old val contains the value stored at the address before the operation, and new val contains the value after the operation. Thus, if old val and new val are the same, it was a read operation. If they are different, it was a write operation.

The AIR constraints needed to support the above structure are as follows.

We still need to make sure memory addresses are contiguous:

(a′−a)⋅(a′−a−1)=0(a' - a) \cdot (a' - a - 1) = 0(a′−a)⋅(a′−a−1)=0

Whenever memory address changes, we want to make sure that old val is set to 000 (i.e., our memory is always initialized to 000). This can be done with the following constraint:

(a′−a)⋅vold′=0(a' - a) \cdot v_{old}' = 0(a′−a)⋅vold′​=0

On the other hand, if memory address doesn't change, we want to make sure that new val in the current row is the same as old val in the next row. This can be done with the following constraint:

(1+a−a′)⋅(vnew−vold′)=0(1 + a - a') \cdot (v_{new} - v_{old}') = 0(1+a−a′)⋅(vnew​−vold′​)=0

Lastly, we need to make sure that for the same address values in clk column are always increasing. One way to do this is to perform a 161616-bit range check on the value of (i′−i−1)(i' - i - 1)(i′−i−1), where iii is the reference to clk column. However, this would mean that memory operations involving the same address must happen within 655366553665536 VM cycles from each other. This limitation would be difficult to enforce statically. To remove this limitation, we need to add two more columns as shown below:

In the above column d0 contains the lower 161616 bits of (i′−i−1)(i' - i - 1)(i′−i−1) while d1 contains the upper 161616 bits. The constraint needed to enforces this is as follows:

(1+a−a′)⋅((i′−i−1)−(216⋅d1′+d0′))=0(1 + a - a') \cdot ((i' - i - 1) - (2^{16} \cdot d_1' + d_0')) = 0(1+a−a′)⋅((i′−i−1)−(216⋅d1′​+d0′​))=0

Additionally, we need to apply 161616-bit range checks to columns d0 and d1.

Overall, the cost of reading or writing a single element is now 666 trace cells and 222 161616-bit range-checks.

Non-contiguous memory​

Requiring that memory addresses are contiguous may also be a difficult limitation to impose statically. To remove this limitation, we need to introduce one more column as shown below:

In the above, the prover sets the value in the new column t to 000 when the address doesn't change, and to 1/(a′−a)1 / (a' - a)1/(a′−a) otherwise. To simplify constraint description, we'll define variable nnn computed as follows:

n=(a′−a)⋅t′n = (a' - a) \cdot t'n=(a′−a)⋅t′

Then, to make sure the prover sets the value of ttt correctly, we'll impose the following constraints:

n2−n=0(1−n)⋅(a′−a)=0n^2 - n = 0 \\ (1 - n) \cdot (a' - a) = 0n2−n=0(1−n)⋅(a′−a)=0

The above constraint, in combination with 161616-bit range checks against columns d0 and d1 ensure that values in addr and clk columns always increase monotonically, and also that column addr may contain duplicates, while values in clk column must be unique for a given address.

Context separation​

In many situations it may be desirable to assign memories to different context. For example, when making a cross-contract calls, memories of the caller and the callee should be separate. That is, caller should not be able to access the memory of the callee and vice-versa.

To accommodate this feature, we need to add one more column as illustrated below.

This new column ctx should behave similarly to the address column: values in it should increase monotonically, and there could be breaks between them. We also need to change how the prover populates column t:

  • If the context changes, t should be set to the inverse (c′−c)(c' - c)(c′−c), where ccc is a reference to column ctx.
  • If the context remains the same but the address changes, column t should be set to the inverse of (a′−a)(a' - a)(a′−a).
  • Otherwise, column t should be set to 000.

To simplify description of constraints, we'll define two variables n0n_0n0​ and n1n_1n1​ as follows:

n0=(c′−c)⋅t′n1=(a′−a)⋅t′n_0 = (c' - c) \cdot t' \\ n_1 = (a' - a) \cdot t'n0​=(c′−c)⋅t′n1​=(a′−a)⋅t′

Thus, n0=1n_0 = 1n0​=1 when the context changes, and 000 otherwise. Also, (1−n0)⋅n1=1(1 - n_0) \cdot n_1 = 1(1−n0​)⋅n1​=1 when context remains the same and address changes, and 000 otherwise.

To make sure the prover sets the value of column t correctly, we'll need to impose the following constraints:

n02−n0=0(1−n0)⋅(c′−c)=0(1−n0)⋅(n12−n1)=0(1−n0)⋅(1−n1)⋅(a′−a)=0n_0^2 - n_0 = 0 \\ (1 - n_0) \cdot (c' - c) = 0 \\ (1 - n_0) \cdot (n_1^2 - n_1) = 0 \\ (1 - n_0) \cdot (1 - n_1) \cdot (a' - a) = 0n02​−n0​=0(1−n0​)⋅(c′−c)=0(1−n0​)⋅(n12​−n1​)=0(1−n0​)⋅(1−n1​)⋅(a′−a)=0

We can then define the following constraints to make sure values in columns d0 and d1 contain the delta between contexts, between addresses, or between clock cycles.

ConditionConstraintComments
n0=1n_0=1n0​=1(c′−c)−(216⋅d1′+d0′)=0(c' - c) - (2^{16} \cdot d_1' + d_0') = 0(c′−c)−(216⋅d1′​+d0′​)=0When the context changes, columns d0 and d1 at the next row should contain the delta between the old and the new contexts.

We can combine the above constraints as follows:

The above constraint, in combination with 161616-bit range checks against columns d0 and d1 ensure that values in ctx, addr, and clk columns always increase monotonically, and also that columns ctx and addr may contain duplicates, while the values in column clk must be unique for a given combination of ctx and addr.

Notice that the above constraint has degree 555.

Miden approach​

While the approach described above works, it comes at significant cost. Reading or writing a single value requires 888 trace cells and 222 161616-bit range checks. Assuming a single range check requires roughly 222 trace cells, the total number of trace cells needed grows to 121212. This is about 666x worse the simple contiguous write-once memory described earlier.

Miden VM frequently needs to deal with batches of 444 field elements, which we call words. For example, the output of Rescue Prime Optimized hash function is a single word. A single 256-bit integer value can be stored as two words (where each element contains one 323232-bit value). Thus, we can optimize for this common use case by making the memory word-addressable. That is 444 field elements are located at each memory address, and we can read and write elements to/from memory in batches of four.

The layout of Miden VM memory table is shown below:

where:

  • s0 is a selector column which is set to 111 for read operations and 000 for write operations.
  • s1 is a selector oclumn which is set to 111 when previously accessed memory is being read and 000 otherwise. In other words, it is set to 111 only when the context and address are the same as they were in the previous row and the s0 operation selector is set to 111 (indicating a read).
  • ctx contains context ID. Values in this column must increase monotonically but there can be gaps between two consecutive values of up to 2322^{32}232. Also, two consecutive values can be the same. In AIR constraint description below, we refer to this column as ccc.
  • addr contains memory address. Values in this column must increase monotonically for a given context but there can be gaps between two consecutive values of up to 2322^{32}232. Also, two consecutive values can be the same. In AIR constraint description below, we refer to this column as aaa.
  • clk contains clock cycle at which the memory operation happened. Values in this column must increase monotonically for a given context and memory address but there can be gaps between two consecutive values of up to 2322^{32}232. In AIR constraint description below, we refer to this column as iii.
  • v0, v1, v2, v3 columns contain field elements stored at a given context/address/clock cycle after the memory operation.
  • Columns d0 and d1 contain lower and upper 161616 bits of the delta between two consecutive context IDs, addresses, or clock cycles. Specifically:
    • When the context changes, these columns contain (c′−c)(c' - c)(c′−c).
    • When the context remains the same but the address changes, these columns contain (a′−a)(a' - a)(a′−a).
    • When both the context and the address remain the same, these columns contain (i′−i−1)(i' - i - 1)(i′−i−1).
  • Column t contains the inverse of the delta between two consecutive context IDs, addresses, or clock cycles. Specifically:
    • When the context changes, this column contains the inverse of (c′−c)(c' - c)(c′−c).
    • When the context remains the same but the address changes, this column contains the inverse of (a′−a)(a' - a)(a′−a).
    • When both the context and the address remain the same, this column contains the inverse of (i′−i−1)(i' - i - 1)(i′−i−1).

For every memory access operation (i.e., read or write), a new row is added to the memory table. For read operations, s0 is set to 111. If neither ctx nor addr have changed, then s1 is set to 111 and the v columns are set to equal the values from the previous row. If ctx or addr have changed, then s1 is set to 000 and the v columns are initialized to 000. For write operations, the values may be different, and both selector columns s0 and s1 are set to 000.

The amortized cost of reading or writing a single value is between 444 and 555 trace cells (this accounts for the trace cells needed for 161616-bit range checks). Thus, from performance standpoint, this approach is roughly 2.52.52.5x worse than the simple contiguous write-once memory described earlier. However, our view is that this trade-off is worth it given that this approach provides read-write memory, context separation, and eliminates the contiguous memory requirement.

AIR constraints​

To simplify description of constraints, we'll define two variables n0n_0n0​ and n1n_1n1​ as follows:

n0=Δc⋅t′n1=Δa⋅t′n_0 = \Delta c \cdot t' \\ n_1 = \Delta a \cdot t'n0​=Δc⋅t′n1​=Δa⋅t′

Where Δc=c′−c\Delta c = c' - cΔc=c′−c and Δa=a′−a\Delta a = a' - aΔa=a′−a.

To make sure the prover sets the value of column t correctly, we'll need to impose the following constraints:

n02−n0=0 | degree=4n_0^2 - n_0 = 0 \text{ | degree} = 4n02​−n0​=0 | degree=4
(1−n0)⋅Δc=0 | degree=3(1 - n_0) \cdot \Delta c = 0 \text{ | degree} = 3(1−n0​)⋅Δc=0 | degree=3
(1−n0)⋅(n12−n1)=0 | degree=6(1 - n_0) \cdot (n_1^2 - n_1) = 0 \text{ | degree} = 6(1−n0​)⋅(n12​−n1​)=0 | degree=6
(1−n0)⋅(1−n1)⋅Δa=0 | degree=5(1 - n_0) \cdot (1 - n_1) \cdot \Delta a = 0 \text{ | degree} = 5(1−n0​)⋅(1−n1​)⋅Δa=0 | degree=5

The above constraints guarantee that when context changes, n0=1n_0 = 1n0​=1. When context remains the same but address changes, (1−n0)⋅n1=1(1 - n_0) \cdot n_1 = 1(1−n0​)⋅n1​=1. And when neither the context nor the address change, (1−n0)⋅(1−n1)=1(1 - n_0) \cdot (1 - n_1) = 1(1−n0​)⋅(1−n1​)=1.

To enforce the values of the selector columns, we first require that they both contain only binary values.

s02−s0=0 | degree=2s_0^2 - s_0 = 0 \text{ | degree} = 2s02​−s0​=0 | degree=2
s12−s1=0 | degree=2s_1^2 - s_1 = 0 \text{ | degree} = 2s12​−s1​=0 | degree=2

Then we require that s1s_1s1​ is always set to 111 during read operations when the context and address did not change and to 000 in all other cases.

(1−n0)⋅(1−n1)⋅s0′⋅(1−s1′)=0 | degree=6(1 - n_0) \cdot (1 - n_1) \cdot s'_0 \cdot (1 - s'_1) = 0 \text{ | degree} = 6(1−n0​)⋅(1−n1​)⋅s0′​⋅(1−s1′​)=0 | degree=6
(n0+(1−n0)⋅n1+(1−s0′))⋅s1′=0 | degree=5(n_0 + (1 - n_0) \cdot n_1 + (1 - s'_0)) \cdot s'_1 = 0 \text{ | degree} = 5(n0​+(1−n0​)⋅n1​+(1−s0′​))⋅s1′​=0 | degree=5

The first constraint enforces that s_1 is 111 when the operation is a read and ctx and addr are both unchanged. The second constraint enforces that when either the context changed, the address changed, or the operation is a write, then s_1 is set to 000.

To enforce the values of context ID, address, and clock cycle grow monotonically as described in the previous section, we define the following constraint.

(n0⋅Δc+(1−n0)⋅(n1⋅Δa+(1−n1)⋅Δi))−(216⋅d1′+d0′)=0 | degree=5\left(n_0 \cdot \Delta c + (1 - n_0) \cdot (n_1 \cdot \Delta a + (1 - n_1) \cdot \Delta i) \right) - (2^{16} \cdot d_1' + d_0') = 0 \text{ | degree} = 5(n0​⋅Δc+(1−n0​)⋅(n1​⋅Δa+(1−n1​)⋅Δi))−(216⋅d1′​+d0′​)=0 | degree=5

Where Δi=i′−i−1\Delta i = i' - i - 1Δi=i′−i−1.

In addition to this constraint, we also need to make sure that values in registers d0d_0d0​ and d1d_1d1​ are less than 2162^{16}216, and this can be done with permutation-based range checks.

Next, we need to make sure that values at a given memory address are always initialized to 000. This can be done with the following constraint:

s0⋅(1−s1)⋅vi=0 for i∈{0,1,2,3} | degree=3s_0 \cdot (1 - s_1) \cdot v_i = 0 \text{ for } i \in \{0, 1, 2, 3\} \text{ | degree} = 3s0​⋅(1−s1​)⋅vi​=0 for i∈{0,1,2,3} | degree=3

Thus, when the operation is a read and either the context changes or the address changes, values in the viv_ivi​ columns are guaranteed to be zeros.

Lastly, we need to make sure that for the same context/address combination, the viv_ivi​ columns of the current row are equal to the corresponding viv_ivi​ columns of the next row. This can be done with the following constraints:

s1⋅(vi′−vi)=0 for i∈{0,1,2,3} | degree=2s_1 \cdot (v_i' - v_i) = 0 \text{ for } i \in \{0, 1, 2, 3\} \text{ | degree} = 2s1​⋅(vi′​−vi​)=0 for i∈{0,1,2,3} | degree=2

Memory row value​

Communication between the memory chiplet and the stack is accomplished via the chiplet bus bchipb_{chip}bchip​. To respond to memory access requests from the stack, we need to divide the current value in bchipb_{chip}bchip​ by the value representing a row in the memory table. This value can be computed as follows:

vmem=α0+α1⋅opmem+α2⋅c+α3⋅a+α4⋅i+∑j=03(αj+5⋅vj)v_{mem} = \alpha_0 + \alpha_1 \cdot op_{mem} + \alpha_2 \cdot c + \alpha_3 \cdot a + \alpha_4 \cdot i + \sum_{j=0}^3(\alpha_{j + 5} \cdot v_j)vmem​=α0​+α1​⋅opmem​+α2​⋅c+α3​⋅a+α4​⋅i+j=0∑3​(αj+5​⋅vj​)

Where, opmemop_{mem}opmem​ is the unique operation label of the memory access operation.

To ensure that values of memory table rows are included into the chiplet bus, we impose the following constraint:

bchip′=bchip⋅vmem | degree=2b_{chip}' = b_{chip} \cdot v_{mem} \text{ | degree} = 2bchip′​=bchip​⋅vmem​ | degree=2

On the stack side, for every memory access request, a corresponding value is divided out of the bchipb_{chip}bchip​ column. Specifics of how this is done are described here.
