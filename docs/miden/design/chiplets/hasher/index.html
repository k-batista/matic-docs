Hash Chiplet

Miden VM "offloads" all hash-related computations to a separate hash processor. This chiplet supports executing the Rescue Prime Optimized hash function (or rather a specific instantiation of it) in the following settings:

  • A single permutation of Rescue Prime Optimized.
  • A simple 2-to-1 hash.
  • A linear hash of nnn field elements.
  • Merkle path verification.
  • Merkle root update.

The chiplet can be thought of as having a small instruction set of 111111 instructions. These instructions are listed below, and examples of how these instructions are used by the chiplet are described in the following sections.

InstructionDescription
HRExecutes a single round of the VM's native hash function. All cycles which are not one less than a multiple of 888 execute this instruction. That is, the chiplet executes this instruction on cycles 0,1,2,3,4,5,60, 1, 2, 3, 4, 5, 60,1,2,3,4,5,6, but not 777, and then again, 8,9,10,11,12,13,148, 9, 10, 11, 12, 13, 148,9,10,11,12,13,14, but not 151515 etc.
BPInitiates computation of a single permutation, a 2-to-1 hash, or a linear hash of many elements. This instruction can be executed only on cycles which are multiples of 888, and it can also be executed concurrently with an HR instruction.
MPInitiates Merkle path verification computation. This instruction can be executed only on cycles which are multiples of 888, and it can also be executed concurrently with an HR instruction.
MVInitiates Merkle path verification for the "old" node value during Merkle root update computation. This instruction can be executed only on cycles which are multiples of 888, and it can also be executed concurrently with an HR instruction.
MUInitiates Merkle path verification for the "new" node value during Merkle root update computation. This instruction can be executed only on cycles which are multiples of 888, and it can also be executed concurrently with an HR instruction.
HOUTReturns the result of the currently running computation. This instruction can be executed only on cycles which are one less than a multiple of 888 (e.g. 777, 151515 etc.).
SOUTReturns the whole hasher state. This instruction can be executed only on cycles which are one less than a multiple of 888, and only if the computation was started using BP instruction.
ABPAbsorbs a new set of elements into the hasher state when computing a linear hash of many elements. This instruction can be executed only on cycles which are one less than a multiple of 888, and only if the computation was started using BP instruction.
MPAAbsorbs the next Merkle path node into the hasher state during Merkle path verification computation. This instruction can be executed only on cycles which are one less than a multiple of 888, and only if the computation was started using MP instruction.
MVAAbsorbs the next Merkle path node into the hasher state during Merkle path verification for the "old" node value during Merkle root update computation. This instruction can be executed only on cycles which are one less than a multiple of 888, and only if the computation was started using MV instruction.
MUAAbsorbs the next Merkle path node into the hasher state during Merkle path verification for the "new" node value during Merkle root update computation. This instruction can be executed only on cycles which are one less than a multiple of 888, and only if the computation was started using Mu instruction.

Chiplet trace​

Execution trace table of the chiplet consists of 171717 trace columns and 333 periodic columns. The structure of the table is such that a single permutation of the hash function can be computed using 888 table rows. The layout of the table is illustrated below.

The meaning of the columns is as follows:

  • Three periodic columns k0k_0k0​, k1k_1k1​, and k2k_2k2​ are used to help select the instruction executed at a given row. All of these columns contain patterns which repeat every 888 rows. For k0k_0k0​ the pattern is 777 zeros followed by 111 one, helping us identify the last row in the cycle. For k1k_1k1​ the pattern is 666 zeros, 111 one, and 111 zero, which can be used to identify the second-to-last row in a cycle. For k2k_2k2​ the pattern is 111 one followed by 777 zeros, which can identify the first row in the cycle.
  • Three selector columns s0s_0s0​, s1s_1s1​, and s2s_2s2​. These columns can contain only binary values (ones or zeros), and they are also used to help select the instruction to execute at a given row.
  • One row address column rrr. This column starts out at 111 and gets incremented by 111 with every row.
  • Twelve hasher state columns h0,...,h11h_0, ..., h_{11}h0​,...,h11​. These columns are used to hold the hasher state for each round of the hash function permutation. The state is laid out as follows:
    • The first four columns (h0,...,h3h_0, ..., h_3h0​,...,h3​) are reserved for capacity elements of the state. When the state is initialized for hash computations, h0h_0h0​ should be set to 000 if the number of elements to be hashed is a multiple of the rate width (888). Otherwise, h0h_0h0​ should be set to 111. h1h_1h1​ should be set to the domain value if a domain has been provided (as in the case of control block hashing). All other capacity elements should be set to 000's.
    • The next eight columns (h4,...,h11h_4, ..., h_{11}h4​,...,h11​) are reserved for the rate elements of the state. These are used to absorb the values to be hashed. Once the permutation is complete, hash output is located in the first four rate columns (h4,...,h7h_4, ..., h_7h4​,...,h7​).
  • One index column iii. This column is used to help with Merkle path verification and Merkle root update computations.

In addition to the columns described above, the chiplet relies on two running product columns which are used to facilitate multiset checks (similar to the ones described here). These columns are:

  • p0p_0p0​ - which is used to tie the chiplet table with the main VM's stack. That is, values representing inputs consumed by the chiplet and outputs produced by the chiplet are multiplied into p0p_0p0​, while the main VM stack divides them out of p0p_0p0​. Thus, if the sets of inputs and outputs between the main VM stack and hash chiplet are the same, value of p0p_0p0​ should be equal to 111 at the start and the end of the execution trace.
  • p1p_1p1​ - which is used to keep track of the sibling table used for Merkle root update computations. Specifically, when a root for the old leaf value is computed, we add an entry for all sibling nodes to the table (i.e., we multiply p1p_1p1​ by the values representing these entries). When the root for the new leaf value is computed, we remove the entries for the nodes from the table (i.e., we divide p1p_1p1​ by the value representing these entries). Thus, if both computations used the same set of sibling nodes (in the same order), the sibling table should be empty by the time Merkle root update procedure completes (i.e., the value of p1p_1p1​ would be 111).

A few additional notes about flag values:

  • With the exception of frprf_{rpr}frpr​, all flags are mutually exclusive. That is, if one flag is set to 111, all other flats are set to 000.
  • With the exception of frprf_{rpr}frpr​, computing flag values involves 333 multiplications, and thus the degree of these flags is 444.
  • We can also define a flag fout=k0⋅(1−s0)⋅(1−s1)f_{out} = k_0 \cdot (1 - s_0) \cdot (1 - s_1)fout​=k0​⋅(1−s0​)⋅(1−s1​). This flag will be set to 111 when either fhout=1f_{hout}=1fhout​=1 or fsout=1f_{sout}=1fsout​=1 in the current row.
  • We can define a flag fout′=k1⋅(1−s0′)⋅(1−s1′)f_{out}' = k_1 \cdot (1 - s_0') \cdot (1 - s_1')fout′​=k1​⋅(1−s0′​)⋅(1−s1′​). This flag will be set to 111 when either fhout=1f_{hout}=1fhout​=1 or fsout=1f_{sout}=1fsout​=1 in the next row.

We also impose the following restrictions on how values in selector columns can be updated:

  • Values in columns s1s_1s1​ and s2s_2s2​ must be copied over from one row to the next, unless fout=1f_{out} = 1fout​=1 or fout′=1f_{out}' = 1fout′​=1 indicating the hout or sout flag is set for the current or the next row.
  • Value in s0s_0s0​ must be set to 111 if fout=1f_{out}=1fout​=1 for the previous row, and to 000 if any of the flags fabpf_{abp}fabp​, fmpaf_{mpa}fmpa​, fmvaf_{mva}fmva​, or fmuaf_{mua}fmua​ are set to 111 for the previous row.

The above rules ensure that we must finish one computation before starting another, and we can't change the type of the computation before the computation is finished.

Computation examples​

Single permutation​

Computing a single permutation of Rescue Prime Optimized hash function involves the following steps:

  1. Initialize hasher state with 121212 field elements.
  2. Apply Rescue Prime Optimized permutation.
  3. Return the entire hasher state as output.

The chiplet accomplishes the above by executing the following instructions:

[BP, HR]                 // init state and execute a hash round (concurrently)
HR HR HR HR HR HR        // execute 6 more hash rounds
SOUT                     // return the entire state as output

In the above {a0,...,a11}\{a_0, ..., a_{11}\}{a0​,...,a11​} is the input state of the hasher, and {b0,...,b11}\{b_0, ..., b_{11}\}{b0​,...,b11​} is the output state of the hasher.

Simple 2-to-1 hash​

Computing a 2-to-1 hash involves the following steps:

  1. Initialize hasher state with 888 field elements, setting the second capacity element to domaindomaindomain if the domain is provided (as in the case of control block hashing) or else 000, and the remaining capacity elements to 000.
  2. Apply Rescue Prime Optimized permutation.
  3. Return elements 4,...,7{4, ..., 7}4,...,7 of the hasher state as output.

The chiplet accomplishes the above by executing the following instructions:

[BP, HR]                 // init state and execute a hash round (concurrently)
HR HR HR HR HR HR        // execute 6 more hash rounds
HOUT                     // return elements 4, 5, 6, 7 of the state as output

Execution trace for this computation would look as illustrated below.

In the above, we compute the following:

{c0,c1,c2,c3}←hash({a0,a1,a2,a3},{b0,b1,b2,b3})\{c_0, c_1, c_2, c_3\} \leftarrow hash(\{a_0, a_1, a_2, a_3\}, \{b_0, b_1, b_2, b_3\}){c0​,c1​,c2​,c3​}←hash({a0​,a1​,a2​,a3​},{b0​,b1​,b2​,b3​})

Linear hash of n elements​

Computing a linear hash of nnn elements consists of the following steps:

  1. Initialize hasher state with the first 888 elements, setting the first capacity register to 000 if nnn is a multiple of the rate width (888) or else 111, and the remaining capacity elements to 000.
  2. Apply Rescue Prime Optimized permutation.
  3. Absorb the next set of elements into the state (up to 888 elements), while keeping capacity elements unchanged.
  4. Repeat steps 2 and 3 until all nnn elements have been absorbed.
  5. Return elements 4,...,7{4, ..., 7}4,...,7 of the hasher state as output.

The chiplet accomplishes the above by executing the following instructions (for hashing 161616 elements):

[BP, HR]                    // init state and execute a hash round (concurrently)
HR HR HR HR HR HR           // execute 6 more hash rounds
ABP                         // absorb the next set of elements into the state
HR HR HR HR HR HR HR        // execute 7 hash rounds
HOUT                        // return elements 4, 5, 6, 7 of the state as output

Execution trace for this computation would look as illustrated below.

In the above, the value absorbed into hasher state between rows 777 and 888 is the delta between values tit_iti​ and sis_isi​. Thus, if we define bi=ti−sib_i = t_i - s_ibi​=ti​−si​ for i∈{0,...,7}i \in \{0, ..., 7\}i∈{0,...,7}, the above computes the following:

{r0,r1,r2,r3}←hash(a0,...,a7,b0,...,b7)\{r_0, r_1, r_2, r_3\} \leftarrow hash(a_0, ..., a_7, b_0, ..., b_7){r0​,r1​,r2​,r3​}←hash(a0​,...,a7​,b0​,...,b7​)

Verify Merkle path​

Verifying a Merkle path involves the following steps:

  1. Initialize hasher state with the leaf and the first node of the path, setting all capacity elements to 000s. a. Also, initialize the index register to the leaf's index value.
  2. Apply Rescue Prime Optimized permutation. a. Make sure the index value doesn't change during this step.
  3. Copy the result of the hash to the next row, and absorb the next node of the Merkle path into the hasher state. a. Remove a single bit from the index, and use it to determine how to place the copied result and absorbed node in the state.
  4. Repeat steps 2 and 3 until all nodes of the Merkle path have been absorbed.
  5. Return elements 4,...,7{4, ..., 7}4,...,7 of the hasher state as output. a. Also, make sure the index value has been reduced to 000.

The chiplet accomplishes the above by executing the following instructions (for Merkle tree of depth 333):

[MP, HR]                    // init state and execute a hash round (concurrently)
HR HR HR HR HR HR           // execute 6 more hash rounds
MPA                         // copy result & absorb the next node into the state
HR HR HR HR HR HR HR        // execute 7 hash rounds
HOUT                        // return elements 4, 5, 6, 7 of the state as output

Suppose we have a Merkle tree as illustrated below. This Merkle tree has 444 leaves, each of which consists of 444 field elements. For example, leaf aaa consists of elements a0,a1,a2,a3a_0, a_1, a_2, a_3a0​,a1​,a2​,a3​, leaf be consists of elements b0,b1,b2,b3b_0, b_1, b_2, b_3b0​,b1​,b2​,b3​ etc.

If we wanted to verify that leaf ddd is in fact in the tree, we'd need to compute the following hashes:

r←hash(e,hash(c,d))r \leftarrow hash(e, hash(c, d))r←hash(e,hash(c,d))

And if r=gr = gr=g, we can be convinced that ddd is in fact in the tree at position 333. Execution trace for this computation would look as illustrated below.

In the above, the prover provides values for nodes ccc and eee non-deterministically.

Update Merkle root​

Updating a node in a Merkle tree (which also updates the root of the tree) can be simulated by verifying two Merkle paths: the path that starts with the old leaf and the path that starts with the new leaf.

Suppose we have the same Merkle tree as in the previous example, and we want to replace node ddd with node d′d'd′. The computations we'd need to perform are:

r←hash(e,hash(c,d))r′←hash(e,hash(c,d′))r \leftarrow hash(e, hash(c, d)) \\ r' \leftarrow hash(e, hash(c, d'))r←hash(e,hash(c,d))r′←hash(e,hash(c,d′))

Then, as long as r=gr = gr=g, and the same values were used for ccc and eee in both computations, we can be convinced that the new root of the tree is r′r'r′.

The chiplet accomplishes the above by executing the following instructions:

// verify the old merkle path
[MV, HR]                    // init state and execute a hash round (concurrently)
HR HR HR HR HR HR           // execute 6 more hash rounds
MPV                         // copy result & absorb the next node into the state
HR HR HR HR HR HR HR        // execute 7 hash rounds
HOUT                        // return elements 4, 5, 6, 7 of the state as output

// verify the new merkle path
[MU, HR]                    // init state and execute a hash round (concurrently)
HR HR HR HR HR HR           // execute 6 more hash rounds
MPU                         // copy result & absorb the next node into the state
HR HR HR HR HR HR HR        // execute 7 hash rounds
HOUT                        // return elements 4, 5, 6, 7 of the state as output

The semantics of MV and MU instructions are similar to the semantics of MP instruction from the previous example (and MVA and MUA are similar to MPA) with one important difference: MV* instructions add the absorbed node (together with its index in the tree) to permutation column p1p_1p1​, while MU* instructions remove the absorbed node (together with its index in the tree) from p1p_1p1​. Thus, if the same nodes were used during both Merkle path verification, the state of p1p_1p1​ should not change. This mechanism is used to ensure that the same internal nodes were used in both computations.

AIR constraints​

When describing AIR constraints, we adopt the following notation: for column xxx, we denote the value in the current row simply as xxx, and the value in the next row of the column as x′x′x′. Thus, all transition constraints described in this note work with two consecutive rows of the execution trace.

Row address constraint​

As mentioned above, row address rrr starts at 111, and is incremented by 111 with every row. The first condition can be enforced with a boundary constraint which specifies r=1r=1r=1 at the first row. The second condition can be enforced via the following transition constraint:

r′−r−1=0 | degree=1r' - r - 1 = 0 \text{ | degree} = 1r′−r−1=0 | degree=1

This constraint should not be applied to the very last row of the hasher execution trace, since we do not want to enforce a value that would conflict with the first row of a subsequent chiplet in the Chiplets module. Therefore we can create a special virtual flag for this constraint using the chip_s0chip\_s_0chip_s0​ selector column from the Chiplets module that selects for the hash chiplet.

The modified row address constraint which should be applied is the following:

(1−chip_s0′)⋅(r′−r−1)=0 | degree=2(1 - chip\_s_0') \cdot (r' - r - 1) = 0 \text{ | degree} = 2(1−chip_s0′​)⋅(r′−r−1)=0 | degree=2

Note: this constraint should also be multiplied by Chiplets module's selector flag s0s_0s0​, as is true for all constraints in this chiplet.

Selector columns constraints​

For selector columns, first we must ensure that only binary values are allowed in these columns. This can be done with the following constraints:

s02−s0=0 | degree=2s12−s1=0 | degree=2s22−s2=0 | degree=2s_0^2 - s_0 = 0 \text{ | degree} = 2 \\ s_1^2 - s_1 = 0 \text{ | degree} = 2 \\ s_2^2 - s_2 = 0 \text{ | degree} = 2s02​−s0​=0 | degree=2s12​−s1​=0 | degree=2s22​−s2​=0 | degree=2

Next, we need to make sure that unless fout=1f_{out}=1fout​=1 or fout′=1f_{out}'=1fout′​=1, the values in columns s1s_1s1​ and s2s_2s2​ are copied over to the next row. This can be done with the following constraints:

(s1′−s1)⋅(1−fout′)⋅(1−fout)=0 | degree=9(s2′−s2)⋅(1−fout′)⋅(1−fout)=0 | degree=9(s_1' - s_1) \cdot (1 - f_{out}') \cdot (1 - f_{out}) = 0 \text{ | degree} = 9 \\ (s_2' - s_2) \cdot (1 - f_{out}') \cdot (1 - f_{out}) = 0 \text{ | degree} = 9(s1′​−s1​)⋅(1−fout′​)⋅(1−fout​)=0 | degree=9(s2′​−s2​)⋅(1−fout′​)⋅(1−fout​)=0 | degree=9

Next, we need to enforce that if any of fabp,fmpa,fmva,fmuaf_{abp}, f_{mpa}, f_{mva}, f_{mua}fabp​,fmpa​,fmva​,fmua​ flags is set to 111, the next value of s0s_0s0​ is 000. In all other cases, s0s_0s0​ should be unconstrained. These flags will only be set for rows that are 1 less than a multiple of 8 (the last row of each cycle). This can be done with the following constraint:

s0′⋅(fabp+fmpa+fmva+fmua)=0 | degree=5s_0' \cdot (f_{abp} + f_{mpa} + f_{mva} + f_{mua})= 0 \text{ | degree} = 5s0′​⋅(fabp​+fmpa​+fmva​+fmua​)=0 | degree=5

Lastly, we need to make sure that no invalid combinations of flags are allowed. This can be done with the following constraints:

k0⋅(1−s0)⋅s1=0 | degree=3k_0 \cdot (1 - s_0) \cdot s_1 = 0 \text{ | degree} = 3k0​⋅(1−s0​)⋅s1​=0 | degree=3

The above constraints enforce that on every step which is one less than a multiple of 888, if s0=0s_0 = 0s0​=0, then s1s_1s1​ must also be set to 000. Basically, if we set s0=0s_0=0s0​=0, then we must make sure that either fhout=1f_{hout}=1fhout​=1 or fsout=1f_{sout}=1fsout​=1.

Node index constraints​

Node index column iii is relevant only for Merkle path verification and Merkle root update computations, but to simplify the overall constraint system, the same constraints will be imposed on this column for all computations.

Overall, we want values in the index column to behave as follows:

  • When we start a new computation, we should be able to set iii to an arbitrary value.
  • When a computation is finished, value in iii must be 000.
  • When we absorb a new node into the hasher state we must shift the value in iii by one bit to the right.
  • In all other cases value in iii should not change.

A shift by one bit to the right can be described with the following equation: i=2⋅i′+bi = 2 \cdot i' + bi=2⋅i′+b, where bbb is the value of the bit which is discarded. Thus, as long as bbb is a binary value, the shift to the right is performed correctly, and this can be enforced with the following constraint:

b2−b=0b^2 - b = 0b2−b=0

Since we want to enforce this constraint only when a new node is absorbed into the hasher state, we'll define a flag for when this should happen as follows:

Hasher state constraints​

Hasher state columns h0,...,h11h_0, ..., h_{11}h0​,...,h11​ should behave as follows:

  • On the 888th row of every 888-row cycle, we apply the constraints based on which transition flag is set as described in the table below.

Specifically, when absorbing the next set of elements into the state during linear hash computation (i.e., fabp=1f_{abp} = 1fabp​=1), the first 444 elements (the capacity portion) are carried over to the next row. For j∈{0,...,3}j \in \{0, ..., 3\}j∈{0,...,3} this can be described as follows:

Note, that when a computation is completed (i.e., fout=1f_{out}=1fout​=1), the next hasher state is unconstrained.

Hash chiplet bus constraints​

As described previously, running product column p0p_0p0​ is used to tie the hash chiplet with the main VM's stack. When receiving inputs from or returning results to the stack, hash chiplet multiplies p0p_0p0​ by their respective values. On other other side, when sending inputs to the hash chiplet or receiving results from the chiplet, the stack divides p0p_0p0​ by their values.

Sibling table constraints​

As mentioned previously, sibling table (represented by running column p1p_1p1​) is used to keep track of sibling nodes used during Merkle root update computation. For this computation, we need to enforce the following rules:

  • When computing the old Merkle root, whenever a new sibling node is absorbed into the hasher state (i.e., fmv+fmva=1f_{mv} + f_{mva} = 1fmv​+fmva​=1), an entry for this sibling should be included into p1p_1p1​.
  • When computing the new Merkle root, whenever a new sibling node is absorbed into the hasher state (i.e., fmu+fmua=1f_{mu} + f_{mua} = 1fmu​+fmua​=1), the entry for this sibling should be removed from p1p_1p1​.

To make sure computation of the old Merkle root is immediately followed by the computation of the new Merkle root, we impose the following constraint:

(fbp+fmp+fmv)⋅(1−p1)=0 | degree=5(f_{bp} + f_{mp} + f_{mv}) \cdot (1 - p_1) = 0 \text{ | degree} = 5(fbp​+fmp​+fmv​)⋅(1−p1​)=0 | degree=5

The above means that whenever we start a new computation which is not the computation of the new Merkle root, the sibling table must be empty. Thus, after the hash chiplet computes the old Merkle root, the only way to clear the table is to compute the new Merkle root.

Together with boundary constraints enforcing that p1=1p_1=1p1​=1 at the first and last rows, the above constraints ensure that if a node was included into p1p_1p1​ as a part of computing the old Merkle root, the same node must be removed from p1p_1p1​ as a part of computing the new Merkle root.
