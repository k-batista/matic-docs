Bitwise Chiplet

In this note we describe how to compute bitwise AND and XOR operations on 32-bit values and the constraints required for proving correct execution.

Assume that aaa and bbb are field elements in a 64-bit prime field. Assume also that aaa and bbb are known to contain values smaller than 2322^{32}232. We want to compute a⊕b→za \oplus b \rightarrow za⊕b→z, where ⊕\oplus⊕ is either bitwise AND or XOR, and zzz is a field element containing the result of the corresponding bitwise operation.

First, observe that we can compute AND and XOR relations for single bit values as follows:

and(a,b)=a⋅band(a, b) = a \cdot band(a,b)=a⋅b
xor(a,b)=a+b−2⋅a⋅bxor(a, b) = a + b - 2 \cdot a \cdot bxor(a,b)=a+b−2⋅a⋅b

To compute bitwise operations for multi-bit values, we will decompose the values into individual bits, apply the operations to single bits, and then aggregate the bitwsie results into the final result.

To perform this operation we will use a table with 11 columns, and computing a single AND or XOR operation will require 8 table rows. We will also rely on two periodic columns as shown below.

In the above, the columns have the following meanings:

  • Periodic columns k0k_0k0​ and k1k_1k1​. These columns contain values needed to switch various constraint on or off. k0k_0k0​ contains a repeating sequence of a single one, followed by seven zeros. k1k_1k1​ contains a repeating sequence of seven ones, followed by a single zero.
  • Input columns aaa and bbb. On the first row of each 8-row cycle, the prover will set values in these columns to the upper 4 bits of the values to which a bitwise operation is to be applied. For all subsequent rows, we will append the next-most-significant 4-bit limb to each value. Thus, by the final row columns aaa and bbb will contain the full input values for the bitwise operation.
  • Columns a0a_0a0​, a1a_1a1​, a2a_2a2​, a3a_3a3​, b0b_0b0​, b1b_1b1​, b2b_2b2​, b3b_3b3​ will contain lower 4 bits of their corresponding values.
  • Output column zpz_pzp​. This column represents the value of column zzz for the prior row. For the first row, it is set to 000.
  • Output column zzz. This column will be used to aggregate the results of bitwise operations performed over columns a0a_0a0​, a1a_1a1​, a2a_2a2​, a3a_3a3​, b0b_0b0​, b1b_1b1​, b2b_2b2​, b3b_3b3​. By the time we get to the last row in each 8-row cycle, this column will contain the final result.

Example​

Let's illustrate the above table on a concrete example. For simplicity, we'll use 16-bit values, and thus, we'll only need 4 rows to complete the operation (rather than 8 for 32-bit values). Let's say a=41851a = 41851a=41851 (b1010_0011_0111_1011) and b=40426b = 40426b=40426 (b1001_1101_1110_1010), then and(a,b)=33130and(a, b) = 33130and(a,b)=33130 (b1000_0001_0110_1010). The table for this computation looks like so:

abx0x1x2x3y0y1y2y3zpz
1090101100108

26152526111001111292070
418514042611010101207033130

Here, in the first row, we set each of the aaa and bbb columns to the value of their most-significant 4-bit limb. The bit columns (a0..a3a_0 .. a_3a0​..a3​ and b0..b3b_0 .. b_3b0​..b3​) in the first row contain the lower 4 bits of their corresponding values (b1010 and b1001). Column zzz contains the result of bitwise AND for the upper 4 bits (b1000), while column zpz_pzp​ contains that result for the prior row.

With every subsequent row, we inject the next-most-significant 4 bits of each value into the bit columns, increase the aaa and bbb columns accordingly, and aggregate the result of bitwise AND into the zzz column, adding it to 242^424 times the value of zzz in the previous row. We set column zpz_pzp​ to be the value of zzz in the prior row. By the time we get to the last row, the zzz column contains the result of the bitwise AND, while columns aaa and bbb contain their original values.

Constraints​

AIR constraints needed to ensure the correctness of the above table are described below.

Selectors​

The Bitwise chiplet supports three operations with the following operation selectors:

  • U32AND: s=0s = 0s=0
  • U32XOR: s=1s = 1s=1

The constraints must require that the selectors be binary and stay the same throughout the cycle:

s2−s=0 | degree=2s^2 - s = 0 \text{ | degree} = 2s2−s=0 | degree=2
k1⋅(s′−s)=0 | degree=2k_1 \cdot (s' - s) = 0 \text{ | degree} = 2k1​⋅(s′−s)=0 | degree=2

Input decomposition​

We need to make sure that inputs aaa and bbb are decomposed correctly into their individual bits. To do this, first, we need to make sure that columns a0a_0a0​, a1a_1a1​, a2a_2a2​, a3a_3a3​, b0b_0b0​, b1b_1b1​, b2b_2b2​, b3b_3b3​, can contain only binary values (000 or 111). This can be accomplished with the following constraints (for iii ranging between 000 and 333):

ai2−ai=0 | degree=2a_i^2 - a_i = 0 \text{ | degree} = 2ai2​−ai​=0 | degree=2
bi2−bi=0 | degree=2b_i^2 - b_i = 0 \text{ | degree} = 2bi2​−bi​=0 | degree=2

Then, we need to make sure that on the first row of every 8-row cycle, the values in the columns aaa and bbb are exactly equal to the aggregation of binary values contained in the individual bit columns aia_iai​, and bib_ibi​. This can be enforced with the following constraints:

k0⋅(a−∑i=03(2i⋅ai))=0 | degree=2k_0 \cdot \left(a - \sum_{i=0}^3(2^i \cdot a_i)\right) = 0 \text{ | degree} = 2k0​⋅(a−i=0∑3​(2i⋅ai​))=0 | degree=2
k0⋅(b−∑i=03(2i⋅bi))=0 | degree=2k_0 \cdot \left(b - \sum_{i=0}^3(2^i \cdot b_i)\right) = 0 \text{ | degree} = 2k0​⋅(b−i=0∑3​(2i⋅bi​))=0 | degree=2

The above constraints enforce that when k0=1k_0 = 1k0​=1, a=∑i=03(2i⋅ai)a = \sum_{i=0}^3(2^i \cdot a_i)a=∑i=03​(2i⋅ai​) and b=∑i=03(2i⋅bi)b = \sum_{i=0}^3(2^i \cdot b_i)b=∑i=03​(2i⋅bi​).

Lastly, we need to make sure that for all rows in an 8-row cycle except for the last one, the values in aaa and bbb columns are increased by the values contained in the individual bit columns aia_iai​ and bib_ibi​. Denoting aaa as the value of column aaa in the current row, and a′a'a′ as the value of column aaa in the next row, we can enforce these conditions as follows:

k1⋅(a′−(a⋅16+∑i=03(2i⋅ai′)))=0 | degree=2k_1 \cdot \left(a' - \left(a \cdot 16 + \sum_{i=0}^3(2^i \cdot a'_i)\right)\right) = 0 \text{ | degree} = 2k1​⋅(a′−(a⋅16+i=0∑3​(2i⋅ai′​)))=0 | degree=2
k1⋅(b′−(b⋅16+∑i=03(2i⋅bi′)))=0 | degree=2k_1 \cdot \left(b' - \left(b \cdot 16 + \sum_{i=0}^3(2^i \cdot b'_i)\right)\right) = 0 \text{ | degree} = 2k1​⋅(b′−(b⋅16+i=0∑3​(2i⋅bi′​)))=0 | degree=2

The above constraints enforce that when k1=1k_1 = 1k1​=1 , a′=16⋅a+∑i=03(2i⋅ai′)a' = 16 \cdot a + \sum_{i=0}^3(2^i \cdot a'_i)a′=16⋅a+∑i=03​(2i⋅ai′​) and b′=16⋅b+∑i=03(2i⋅bi′)b' = 16 \cdot b + \sum_{i=0}^3(2^i \cdot b'_i)b′=16⋅b+∑i=03​(2i⋅bi′​).

Output aggregation​

To ensure correct aggregation of operations over individual bits, first we need to ensure that in the first row, the aggregated output value of the previous row should be 0.

k0⋅zp=0 | degree=2k_0 \cdot z_p = 0 \text{ | degree} = 2k0​⋅zp​=0 | degree=2

Next, we need to ensure that for each row except the last, the aggregated output value must equal the previous aggregated output value in the next row.

k1⋅(z−zp′)=0 | degree=2k_1 \cdot \left(z - z'_p\right) = 0 \text{ | degree} = 2k1​⋅(z−zp′​)=0 | degree=2

Lastly, we need to ensure that for all rows the value in the zzz column is computed by multiplying the previous output value (from the zpz_pzp​ column in the current row) by 16 and then adding it to the bitwise operation applied to the row's set of bits of aaa and bbb. The entire constraint must also be multiplied by the operation selector flag to ensure it is only applied for the appropriate operation.

For U32AND, this is enforced with the following constraint:

(1−s)⋅(z−(zp⋅16+∑i=03(2i⋅ai⋅bi)))=0 | degree=3(1 - s) \cdot \left(z -(z_p \cdot 16 + \sum_{i=0}^3(2^i \cdot a_i \cdot b_i))\right) = 0 \text{ | degree} = 3(1−s)⋅(z−(zp​⋅16+i=0∑3​(2i⋅ai​⋅bi​)))=0 | degree=3

For U32XOR, this is enforced with the following constraint:

s⋅(z−(zp⋅16+∑i=03(2i⋅(ai+bi−2⋅ai⋅bi))))=0 | degree=3s \cdot \left(z -(z_p \cdot 16 + \sum_{i=0}^3(2^i \cdot (a_i + b_i - 2 \cdot a_i \cdot b_i)))\right) = 0 \text{ | degree} = 3s⋅(z−(zp​⋅16+i=0∑3​(2i⋅(ai​+bi​−2⋅ai​⋅bi​))))=0 | degree=3

Bitwise chiplet bus constraints​

To simplify the notation for describing bitwise constraints on the chiplet bus, we'll first define variable uuu, which represents how aaa, bbb, and zzz in the execution trace are reduced to a single value. Denoting the random values received from the verifier as α0,α1\alpha_0, \alpha_1α0​,α1​, etc., this can be achieved as follows.

u=α1⋅a+α2⋅b+α3⋅zu = \alpha_1 \cdot a + \alpha_2 \cdot b + \alpha_3 \cdot zu=α1​⋅a+α2​⋅b+α3​⋅z

To request a bitwise operation, the prover will provide the values of aaa, bbb, and zzz non-deterministically to the stack (the component that makes bitwise requests). The lookup can then be performed by dividing (α0+u)\left(\alpha_0 + u\right)(α0​+u) out of the bus column:

bchip′⋅(α0+u)=bchipb'_{chip} \cdot \left(\alpha_0 + u\right) = b_{chip}bchip′​⋅(α0​+u)=bchip​

To provide the results of bitwise operations to the chiplets bus, we want to include values of aaa, bbb and zzz at the last row of the cycle.

First, we'll define another intermediate variable viv_ivi​. It will include uuu into the product when k1=0k_1 = 0k1​=0. (uiu_iui​ represents the value of uuu for row iii of the trace.)

vi=(1−k1)⋅uiv_i = (1-k_1) \cdot u_ivi​=(1−k1​)⋅ui​

Then, setting m=1−k1m = 1 - k_1m=1−k1​, we can compute the permutation product from the bitwise chiplet as follows:

∏i=0n((α0+vi)⋅mi+1−mi)\prod_{i=0}^n ((\alpha_0 + v_i) \cdot m_i + 1 - m_i)i=0∏n​((α0​+vi​)⋅mi​+1−mi​)

The above ensures that when 1−k1=01 - k_1 = 01−k1​=0 (which is true for all rows in the 8-row cycle except for the last one), the product does not change. Otherwise, (α0+vi)(\alpha_0 + v_i)(α0​+vi​) gets included into the product.

The constraints for the two sides of the bus communication are combined as follows:

bchip′⋅(α0+ui)=bchip⋅((α0+vi)⋅mi+1−mi) | degree=4b'_{chip} \cdot \left(\alpha_0 + u_i\right) = b_{chip} \cdot ((\alpha_0 + v_i) \cdot m_i + 1 - m_i) \text{ | degree} = 4bchip′​⋅(α0​+ui​)=bchip​⋅((α0​+vi​)⋅mi​+1−mi​) | degree=4
