AIR Constraints for Miden VM

In this section we describe AIR constraint for Miden VM program decoder. These constraints enforce that the execution trace generated by the prover when executing a particular program complies with the rules described in the previous section.

To refer to decoder execution trace columns, we use the names shown on the diagram below (these are the same names as in the previous section). Additionally, we denote the register containing the value at the top of the stack as s0s_0s0​.

We assume that the VM exposes a flag per operation which is set to 111 when the operation is executed, and to 000 otherwise. The notation for such flags is fopnamef_{opname}fopname​. For example, when the VM executes a PUSH operation, flag fpush=1f_{push} = 1fpush​=1. All flags are mutually exclusive - i.e., when one flag is set to 111 all other flags are set to 000. The flags are computed based on values in op_bits columns.

AIR constraints for the decoder involve operations listed in the table below. For each operation we also provide the degree of the corresponding flag and the effect that the operation has on the operand stack (however, in this section we do not cover the constraints needed to enforce the correct transition of the operand stack).

OperationFlagDegreeEffect on stack
JOINfjoinf_{join}fjoin​6Stack remains unchanged.
SPLITfsplitf_{split}fsplit​6Top stack element is dropped.
LOOPfloopf_{loop}floop​6Top stack element is dropped.
REPEATfrepeatf_{repeat}frepeat​4Top stack element is dropped.
SPANfspanf_{span}fspan​6Stack remains unchanged.
RESPANfrespanf_{respan}frespan​4Stack remains unchanged.
CALLfcallf_{call}fcall​4Top stack element is dropped.
SYSCALLfsyscallf_{syscall}fsyscall​4Top stack element is dropped.
ENDfendf_{end}fend​4When exiting a loop block, top stack element is dropped; otherwise, the stack remains unchanged.
HALTfhaltf_{halt}fhalt​4Stack remains unchanged.
PUSHfpushf_{push}fpush​4An immediate value is pushed onto the stack.

We also use the control flow flag fctrlf_{ctrl}fctrl​ exposed by the VM, which is set when any one of the above control flow operations is being executed. It has degree 444.

As described previously, the general idea of the decoder is that the prove provides the program to the VM by populating some of cells in the trace non-deterministically. Values in these are then used to update virtual tables (represented via multiset checks) such as block hash table, block stack table etc. Transition constraints are used to enforce that the tables are updates correctly, and we also apply boundary constraints to enforce the correct initial and final states of these tables. One of these boundary constraints binds the execution trace to the hash of the program being executed. Thus, if the virtual tables were updated correctly and boundary constraints hold, we can be convinced that the prover executed the claimed program on the VM.

In the sections below, we describe constraints according to their logical grouping. However, we start out with a set of general constraints which are applicable to multiple parts of the decoder.

General constraints​

When SPLIT or LOOP operation is executed, the top of the operand stack must contain a binary value:

(fsplit+floop)⋅(s02−s0)=0 | degree=8(f_{split} + f_{loop}) \cdot (s_0^2 - s_0) = 0 \text{ | degree} = 8(fsplit​+floop​)⋅(s02​−s0​)=0 | degree=8

When REPEAT operation is executed, the value at the top of the operand stack must be 111:

frepeat⋅(1−s0)=0 | degree=5f_{repeat} \cdot (1 - s_0) = 0 \text{ | degree} = 5frepeat​⋅(1−s0​)=0 | degree=5

Also, when REPEAT operation is executed, the value in h4h_4h4​ column (the is_loop_body flag), must be set to 111. This ensures that REPEAT operation can be executed only inside a loop:

frepeat⋅(1−h4)=0 | degree=5f_{repeat} \cdot (1 - h_4) = 0 \text{ | degree} = 5frepeat​⋅(1−h4​)=0 | degree=5

When RESPAN operation is executed, we need to make sure that the block ID is incremented by 888:

frespan⋅(a′−a−8)=0 | degree=5f_{respan} \cdot (a' - a - 8) = 0 \text{ | degree} = 5frespan​⋅(a′−a−8)=0 | degree=5

When END operation is executed and we are exiting a loop block (i.e., is_loop, value which is stored in h5h_5h5​, is 111), the value at the top of the operand stack must be 000:

fend⋅h5⋅s0=0 | degree=6f_{end} \cdot h_5 \cdot s_0 = 0 \text{ | degree} = 6fend​⋅h5​⋅s0​=0 | degree=6

Also, when END operation is executed and the next operation is REPEAT, values in h0,...,h4h_0, ..., h_4h0​,...,h4​ (the hash of the current block and the is_loop_body flag) must be copied to the next row:

fend⋅frepeat′⋅(hi′−hi)=0 for i∈0..4 | degree=9f_{end} \cdot f_{repeat}' \cdot (h_i' - h_i) = 0 \text { for } i \in 0..4 \text{ | degree} = 9fend​⋅frepeat′​⋅(hi′​−hi​)=0 for i∈0..4 | degree=9

A HALT instruction can be followed only by another HALT instruction:

fhalt⋅(1−fhalt′)=0 | degree=8f_{halt} \cdot (1 - f_{halt}') = 0 \text{ | degree} = 8fhalt​⋅(1−fhalt′​)=0 | degree=8

When a HALT operation is executed, block address column must be 000:

fhalt⋅a=0 | degree=5f_{halt} \cdot a = 0 \text{ | degree} = 5fhalt​⋅a=0 | degree=5

Values in op_bits columns must be binary (i.e., either 111 or 000):

bi2−bi=0 for i∈0..6 | degree=2b_i^2 - b_i = 0 \text{ for } i \in 0..6 \text{ | degree} = 2bi2​−bi​=0 for i∈0..6 | degree=2

When the value in is_span column is set to 111, control flow operations cannot be executed on the VM, but when is_span flag is 000, only control flow operations can be executed on the VM:

1−sp−fctrl=0 | degree=41 - sp - f_{ctrl} = 0 \text{ | degree} = 41−sp−fctrl​=0 | degree=4

Block hash computation constraints​

As described previously, when the VM starts executing a new block, it also initiates computation of the block's hash. There are two separate methodologies for computing block hashes.

For join, split, and loop blocks, the hash is computed directly from the hashes of the block's children. The prover provides these child hashes non-deterministically by populating registers h0,...,h7h_0,..., h_7h0​,...,h7​. The hasher is initialized using the hash chiplet, and we use the address of the hasher as the block's ID. The result of the hash is available 777 rows down in the hasher table (i.e., at row with index equal to block ID plus 777). We read the result from the hasher table at the time the END operation is executed for a given block.

For span blocks, the hash is computed by absorbing a linear sequence of instructions (organized into operation groups and batches) into the hasher and then returning the result. The prover provides operation batches non-deterministically by populating registers h0,...,h7h_0, ..., h_7h0​,...,h7​. Similarly to other blocks, the hasher is initialized using the hash chiplet at the start of the block, and we use the address of the hasher as the ID of the first operation batch in the block. As we absorb additional operation batches into the hasher (by executing RESPAN operation), the batch address is incremented by 888. This moves the "pointer" into the hasher table 888 rows down with every new batch. We read the result from the hasher table at the time the END operation is executed for a given block.

The way the decoder communicates with the hash chiplet is by dividing values of the multiset check column p0p_0p0​ by the values of operations providing inputs to or reading outputs from the hash chiplet. A constraint to enforce this would look as p0′⋅u=p0p_0' \cdot u = p_0p0′​⋅u=p0​, where uuu is the value which defines the operation.

In constructing value of uuu for decoder AIR constraints, we will use the following labels (see here for an explanation of how values for these labels are computed):

  • mbpm_{bp}mbp​ this label specifies that we are starting a new hash computation.
  • mabpm_{abp}mabp​ this label specifies that we are absorbing the next sequence of 888 elements into an ongoing hash computation.
  • mhoutm_{hout}mhout​ this label specifies that we are reading the result of a hash computation.

To simplify constraint description, we define the following variables:

hinit=α0+α1⋅mbp+α2⋅a′+∑i=07(αi+8⋅hi)h_{init} = \alpha_0 + \alpha_1 \cdot m_{bp} + \alpha_2 \cdot a' + \sum_{i=0}^7(\alpha_{i + 8} \cdot h_i)hinit​=α0​+α1​⋅mbp​+α2​⋅a′+i=0∑7​(αi+8​⋅hi​)

In the above, hinith_{init}hinit​ can be thought of as initiating a hasher with address a′a'a′ and absorbing 888 elements from the hasher state (h0,...,h7h_0, ..., h_7h0​,...,h7​) into it. Control blocks are always padded to fill the hasher rate and as such the α4\alpha_4α4​ (first capacity register) term is set to 000.

habp=α0+α1⋅mabp+α2⋅a′+∑i=07(αi+8⋅hi)h_{abp} = \alpha_0 + \alpha_1 \cdot m_{abp} + \alpha_2 \cdot a' + \sum_{i=0}^7(\alpha_{i + 8} \cdot h_i)habp​=α0​+α1​⋅mabp​+α2​⋅a′+i=0∑7​(αi+8​⋅hi​)

It should be noted that aaa refers to a column in the decoder, as depicted. The addresses in this column are set using the address from the hasher chiplet for the corresponding hash initialization / absorption / return. In the case of habph_{abp}habp​ the value of the address in column aaa in the current row of the decoder is set to equal the value of the address of the row in the hasher chiplet where the previous absorption (or initialization) occurred. a′a'a′ is the address of the next row of the decoder, which is set to equal the address in the hasher chiplet where the absorption referred to by the habph_{abp}habp​ label is happening.

hres=α0+α1⋅mhout+α2⋅(a+7)+∑i=03(αi+8⋅hi)h_{res} = \alpha_0 + \alpha_1 \cdot m_{hout} + \alpha_2 \cdot (a + 7) + \sum_{i=0}^3(\alpha_{i + 8} \cdot h_i)hres​=α0​+α1​⋅mhout​+α2​⋅(a+7)+i=0∑3​(αi+8​⋅hi​)

In the above, aaa represents the address value in the decoder which corresponds to the hasher chiplet address at which the hasher was initialized (or the last absorption took place). As such, a+7a + 7a+7 corresponds to the hasher chiplet address at which the result is returned.

fctrli=fjoin+fsplit+floop+fcall+fsyscall | degree=6f_{ctrli} = f_{join} + f_{split} + f_{loop} + f_{call} + f_{syscall} \text{ | degree} = 6fctrli​=fjoin​+fsplit​+floop​+fcall​+fsyscall​ | degree=6

In the above, fctrlif_{ctrli}fctrli​ is set to 111 when a control flow operation that signifies the initialization of a control block is being executed on the VM. Otherwise, it is set to 000.

d=∑b=06(bi⋅2i)d = \sum_{b=0}^6(b_i \cdot 2^i)d=b=0∑6​(bi​⋅2i)

In the above, ddd represents the opcode value of the opcode being executed on the virtual machine. It is calculated via a bitwise combination of the op bits. We leverage the opcode value to achieve domain separation when hashing control blocks. This is done by populating the second capacity register of the hasher with the value ddd via the α5\alpha_5α5​ term when initializing the hasher.

Using the above variables, we define operation values as described below.

When a control block initializer operation (JOIN, SPLIT, LOOP, CALL, SYSCALL) is executed, a new hasher is initialized and the contents of h0,...,h7h_0, ..., h_7h0​,...,h7​ are absorbed into the hasher. As mentioned above, the opcode value ddd is populated in the second capacity resister via the α5\alpha_5α5​ term.

uctrli=fctrli⋅(hinit+α5⋅d) | degree=7u_{ctrli} = f_{ctrli} \cdot (h_{init} + \alpha_5 \cdot d) \text{ | degree} = 7uctrli​=fctrli​⋅(hinit​+α5​⋅d) | degree=7

When SPAN operation is executed, a new hasher is initialized and contents of h0,...,h7h_0, ..., h_7h0​,...,h7​ are absorbed into the hasher. The number of operation groups to be hashed is padded to a multiple of the rate width (888) and so the α4\alpha_4α4​ is set to 0:

uspan=fspan⋅hinit | degree=7u_{span} = f_{span} \cdot h_{init} \text{ | degree} = 7uspan​=fspan​⋅hinit​ | degree=7

When RESPAN operation is executed, contents of h0,...,h7h_0, ..., h_7h0​,...,h7​ (which contain the new operation batch) are absorbed into the hasher:

urespan=frespan⋅habp | degree=5u_{respan} = f_{respan} \cdot h_{abp} \text{ | degree} = 5urespan​=frespan​⋅habp​ | degree=5

When END operation is executed, the hash result is copied into registers h0,..,h3h_0, .., h_3h0​,..,h3​:

uend=fend⋅hres | degree=5u_{end} = f_{end} \cdot h_{res} \text{ | degree} = 5uend​=fend​⋅hres​ | degree=5

Using the above definitions, we can describe the constraint for computing block hashes as follows:

p0′⋅(uctrli+uspan+urespan+uend+1−(fctrli+fspan+frespan+fend))=p0p_0' \cdot (u_{ctrli} + u_{span} + u_{respan} + u_{end} + \\ 1 - (f_{ctrli} + f_{span} + f_{respan} + f_{end})) = p_0p0′​⋅(uctrli​+uspan​+urespan​+uend​+1−(fctrli​+fspan​+frespan​+fend​))=p0​

We need to add 111 and subtract the sum of the relevant operation flags to ensure that when none of the flags is set to 111, the above constraint reduces to p0′=p0p_0' = p_0p0′​=p0​.

The degree of this constraint is 888.

Block stack table constraints​

As described previously, block stack table keeps track of program blocks currently executing on the VM. Thus, whenever the VM starts executing a new block, an entry for this block is added to the block stack table. And when execution of a block completes, it is removed from the block stack table.

Adding and removing entries to/from the block stack table is accomplished as follows:

  • To add an entry, we multiply the value in column p1p_1p1​ by a value representing a tuple (blk_id, prnt_id, is_loop). A constraint to enforce this would look as p1′=p1⋅vp_1' = p_1 \cdot vp1′​=p1​⋅v, where vvv is the value representing the row to be added.
  • To remove an entry, we divide the value in column p1p_1p1​ by a value representing a tuple (blk_id, prnt_id, is_loop). A constraint to enforce this would look as p1′⋅u=p1p_1' \cdot u = p_1p1′​⋅u=p1​, where uuu is the value representing the row to be removed.

Before describing the constraints for the block stack table, we first describe how we compute the values to be added and removed from the table for each operation. In the below, for block start operations (JOIN, SPLIT, LOOP, SPAN) aaa refers to the ID of the parent block, and a′a'a′ refers to the ID of the starting block. For END operation, the situation is reversed: aaa is the ID of the ending block, and a′a'a′ is the ID of the parent block. For RESPAN operation, aaa refers to the ID of the current operation batch, a′a'a′ refers to the ID of the next batch, and the parent ID for both batches is set by the prover non-deterministically in register h1h_1h1​.

When JOIN operation is executed, row (a′,a,0)(a', a, 0)(a′,a,0) is added to the block stack table:

vjoin=fjoin⋅(α0+α1⋅a′+α2⋅a) | degree=7v_{join} = f_{join} \cdot (\alpha_0 + \alpha_1 \cdot a' + \alpha_2 \cdot a) \text{ | degree} = 7vjoin​=fjoin​⋅(α0​+α1​⋅a′+α2​⋅a) | degree=7

When SPLIT operation is executed, row (a′,a,0)(a', a, 0)(a′,a,0) added to the block stack table:

vsplit=fsplit⋅(α0+α1⋅a′+α2⋅a) | degree=7v_{split} = f_{split} \cdot (\alpha_0 + \alpha_1 \cdot a' + \alpha_2 \cdot a) \text{ | degree} = 7vsplit​=fsplit​⋅(α0​+α1​⋅a′+α2​⋅a) | degree=7

When LOOP operation is executed, row (a′,a,1)(a', a, 1)(a′,a,1) is added to the block stack table if the value at the top of the operand stack is 111, and row (a′,a,0)(a', a, 0)(a′,a,0) is added to the block stack table if the value at the top of the operand stack is 000:

vloop=floop⋅(α0+α1⋅a′+α2⋅a+α3⋅s0) | degree=7v_{loop} = f_{loop} \cdot (\alpha_0 + \alpha_1 \cdot a' + \alpha_2 \cdot a + \alpha_3 \cdot s_0) \text{ | degree} = 7vloop​=floop​⋅(α0​+α1​⋅a′+α2​⋅a+α3​⋅s0​) | degree=7

When SPAN operation is executed, row (a′,a,0)(a', a, 0)(a′,a,0) is added to the block stack table:

vspan=fspan⋅(α0+α1⋅a′+α2⋅a) | degree=7v_{span} = f_{span} \cdot (\alpha_0 + \alpha_1 \cdot a' + \alpha_2 \cdot a) \text{ | degree} = 7vspan​=fspan​⋅(α0​+α1​⋅a′+α2​⋅a) | degree=7

When RESPAN operation is executed, row (a,h1′,0)(a, h_1', 0)(a,h1′​,0) is removed from the block stack table, and row (a′,h1′,0)(a', h_1', 0)(a′,h1′​,0) is added to the table. The prover sets the value of register h1h_1h1​ at the next row to the ID of the parent block:

urespan=frespan⋅(α0+α1⋅a+α2⋅h1′) | degree=5vrespan=frespan⋅(α0+α1⋅a′+α2⋅h1′) | degree=5u_{respan} = f_{respan} \cdot (\alpha_0 + \alpha_1 \cdot a + \alpha_2 \cdot h_1') \text{ | degree} = 5 \\ v_{respan} = f_{respan} \cdot (\alpha_0 + \alpha_1 \cdot a' + \alpha_2 \cdot h_1') \text{ | degree} = 5urespan​=frespan​⋅(α0​+α1​⋅a+α2​⋅h1′​) | degree=5vrespan​=frespan​⋅(α0​+α1​⋅a′+α2​⋅h1′​) | degree=5

When END operation is executed, row (a,a′,h5)(a, a', h_5)(a,a′,h5​) is removed from the block span table. Register h5h_5h5​ contains the is_loop flag:

uend=fend⋅(α0+α1⋅a+α2⋅a′+α3⋅h5) | degree=5u_{end} = f_{end} \cdot (\alpha_0 + \alpha_1 \cdot a + \alpha_2 \cdot a' + \alpha_3 \cdot h_5) \text{ | degree} = 5uend​=fend​⋅(α0​+α1​⋅a+α2​⋅a′+α3​⋅h5​) | degree=5

Using the above definitions, we can describe the constraint for updating the block stack table as follows:

p1′⋅(uend+urespan+1−(fend+frespan))=p1⋅(vjoin+vsplit+vloop+vspan+vrespan+1−(fjoin+fsplit+floop+fspan+frespan))p_1' \cdot (u_{end} + u_{respan} + 1 - (f_{end} + f_{respan})) = \\ p_1 \cdot (v_{join} + v_{split} + v_{loop} + v_{span} + v_{respan} + 1 - (f_{join} + f_{split} + f_{loop} + f_{span} + f_{respan} ))p1′​⋅(uend​+urespan​+1−(fend​+frespan​))=p1​⋅(vjoin​+vsplit​+vloop​+vspan​+vrespan​+1−(fjoin​+fsplit​+floop​+fspan​+frespan​))

We need to add 111 and subtract the sum of the relevant operation flags from each side to ensure that when none of the flags is set to 111, the above constraint reduces to p1′=p1p_1' = p_1p1′​=p1​.

The degree of this constraint is 888.

In addition to the above transition constraint, we also need to impose boundary constraints against the p1p_1p1​ column to make sure the first and the last value in the column is set to 111. This enforces that the block stack table starts and ends in an empty state.

Block hash table constraints​

As described previously, when the VM starts executing a new program block, it adds hashes of the block's children to the block hash table. And when the VM finishes executing a block, it removes the block's hash from the block hash table. This means that the block hash tables gets updated when we execute JOIN, SPLIT, LOOP, REPEAT, and END operations (executing SPAN operation does not affect the block hash table because a span block has no children).

Adding and removing entries to/from the block hash table is accomplished as follows:

  • To add an entry, we multiply the value in column p2p_2p2​ by a value representing a tuple (prnt_id, block_hash, is_first_child, is_loop_body). A constraint to enforce this would look as p2′=p2⋅vp_2' = p_2 \cdot vp2′​=p2​⋅v, where vvv is the value representing the row to be added.
  • To remove an entry, we divide the value in column p2p_2p2​ by a value representing a tuple (prnt_id, block_hash, is_first_child, is_loop_body). A constraint to enforce this would look as p2′⋅u=p2p_2' \cdot u = p_2p2′​⋅u=p2​, where uuu is the value representing the row to be removed.

To simplify constraint descriptions, we define values representing left and right children of a block as follows:

ch1=α0+α1⋅a′+∑i=03(ai+2⋅hi) | degree=1ch2=α0+α1⋅a′+∑i=03(ai+2⋅hi+4) | degree=1ch_1 = \alpha_0 + \alpha_1 \cdot a' + \sum_{i=0}^3(a_{i+2} \cdot h_i) \text{ | degree} = 1 \\ ch_2 = \alpha_0 + \alpha_1 \cdot a' + \sum_{i=0}^3(a_{i+2} \cdot h_{i+4}) \text{ | degree} = 1ch1​=α0​+α1​⋅a′+i=0∑3​(ai+2​⋅hi​) | degree=1ch2​=α0​+α1​⋅a′+i=0∑3​(ai+2​⋅hi+4​) | degree=1

In a similar manner, we define a value representing the result of hash computation as follows:

bh=α0+α1⋅a+∑i=03(αi+2⋅hi)+α7⋅h4 | degree=1bh = \alpha_0 + \alpha_1 \cdot a + \sum_{i=0}^3(\alpha_{i+2} \cdot h_i) + \alpha_7 \cdot h_4 \text{ | degree} = 1bh=α0​+α1​⋅a+i=0∑3​(αi+2​⋅hi​)+α7​⋅h4​ | degree=1

Note that in the above we use aaa (block address from the current row) rather than a′a'a′ (block address from the next row) as we did for for values of ch1ch_1ch1​ and ch2ch_2ch2​. Also, note that we are not adding a flag indicating whether the block is the first child of a join block (i.e., α6\alpha_6α6​ term is missing). It will be added later on.

Using the above variables, we define row values to be added to and removed from the block hash table as follows.

When JOIN operation is executed, hashes of both child nodes are added to the block hash table. We add α6\alpha_6α6​ term to the first child value to differentiate it from the second child (i.e., this sets is_first_child to 111):

vjoin=fjoin⋅(ch1+α6)⋅ch2 | degree=8v_{join} = f_{join} \cdot (ch_1 + \alpha^6) \cdot ch_2 \text{ | degree} = 8vjoin​=fjoin​⋅(ch1​+α6)⋅ch2​ | degree=8

When SPLIT operation is executed and the top of the stack is 111, hash of the true branch is added to the block hash table, but when the top of the stack is 000, hash of the false branch is added to the block hash table:

vsplit=fsplit⋅(s0⋅ch1+(1−s0)⋅ch2) | degree=8v_{split} = f_{split} \cdot (s_0 \cdot ch_1 + (1 - s_0) \cdot ch_2) \text{ | degree} = 8vsplit​=fsplit​⋅(s0​⋅ch1​+(1−s0​)⋅ch2​) | degree=8

When LOOP operation is executed and the top of the stack is 111, hash of loop body is added to the block hash table. We add α7\alpha_7α7​ term to indicate that the child is a body of a loop. The below also means that if the top of the stack is 000, nothing is added to the block hash table as the expression evaluates to 000:

vloop=floop⋅s0⋅(ch1+α7) | degree=8v_{loop} = f_{loop} \cdot s_0 \cdot (ch_1 + \alpha_7) \text{ | degree} = 8vloop​=floop​⋅s0​⋅(ch1​+α7​) | degree=8

When REPEAT operation is executed, hash of loop body is added to the block hash table. We add α7\alpha_7α7​ term to indicate that the child is a body of a loop:

vrepeat=frepeat⋅(ch1+α7) | degree=5v_{repeat} = f_{repeat} \cdot (ch_1 + \alpha_7) \text{ | } \text{degree} = 5vrepeat​=frepeat​⋅(ch1​+α7​) | degree=5

When END operation is executed, hash of the completed block is removed from the block hash table. However, we also need to differentiate between removing the first and the second child of a join block. We do this by looking at the next operation. Specifically, if the next operation is neither END nor REPEAT we know that another block is about to be executed, and thus, we have just finished executing the first child of a join block. Thus, if the next operation is neither END nor REPEAT we need to set the term for α6\alpha_6α6​ coefficient to 111 as shown below:

uend=fend⋅(bh+α6⋅(1−(fend′+frepeat′))) | degree=8u_{end} = f_{end} \cdot (bh + \alpha_6 \cdot (1 - (f_{end}' + f_{repeat}'))) \text{ | } \text{degree} = 8uend​=fend​⋅(bh+α6​⋅(1−(fend′​+frepeat′​))) | degree=8

Using the above definitions, we can describe the constraint for updating the block hash table as follows:

p2′⋅(uend+1−fend)=p2⋅(vjoin+vsplit+vloop+vrepeat+1−(fjoin+fsplit+floop+frepeat))p_2' \cdot (u_{end} + 1 - f_{end}) = \\ p_2 \cdot (v_{join} + v_{split} + v_{loop} + v_{repeat} + 1 - (f_{join} + f_{split} + f_{loop} + f_{repeat}))p2′​⋅(uend​+1−fend​)=p2​⋅(vjoin​+vsplit​+vloop​+vrepeat​+1−(fjoin​+fsplit​+floop​+frepeat​))

We need to add 111 and subtract the sum of the relevant operation flags from each side to ensure that when none of the flags is set to 111, the above constraint reduces to p2′=p2p_2' = p_2p2′​=p2​.

The degree of this constraint is 999.

In addition to the above transition constraint, we also need to set the following boundary constraints against the p2p_2p2​ column:

  • The first value in the column represents a row for the entire program. Specifically, the row tuple would be (0, program_hash, 0, 0). This row should be removed from the table when the last END operation is executed.
  • The last value in the column is 111 - i.e., the block hash table is empty.

Span block​

Span block constraints ensure proper decoding of span blocks. In addition to the block stack table constraints and block hash table constraints described previously, decoding of span blocks requires constraints described below.

In-span column constraints​

The in_span column (denoted as spspsp) is used to identify rows which execute non-control flow operations. The values in this column are set as follows:

  • Executing a SPAN operation sets the value of in_span column to 111.
  • The value remains 111 until the END operation is executed.
  • If RESPAN operation is executed between SPAN and END operations, in the row at which RESPAN operation is executed in_span is set to 000. It is then reset to 111 in the following row.
  • In all other cases, value in the in_span column should be 000.

The picture below illustrates the above rules.

To enforce the above rules we need the following constraints.

When executing SPAN or RESPAN operation, the next value in spspsp column must be set to 111:

(fspan+frespan)⋅(1−sp′)=0 | degree=7(f_{span} + f_{respan}) \cdot (1 - sp') = 0 \text{ | degree} = 7(fspan​+frespan​)⋅(1−sp′)=0 | degree=7

When the next operation is END or RESPAN, the next value in spspsp column must be set 000.

(fend′+frespan′)⋅sp′=0 | degree=5(f_{end}' + f_{respan}') \cdot sp' = 0 \text{ | degree} = 5(fend′​+frespan′​)⋅sp′=0 | degree=5

In all other cases, the value in spspsp column must be copied over to the next row:

(1−fspan−frespan−fend′−frespan′)⋅(sp′−sp)=0 | degree=7(1 - f_{span} - f_{respan} - f_{end}' - f_{respan}') \cdot (sp' - sp) = 0 \text{ | degree} = 7(1−fspan​−frespan​−fend′​−frespan′​)⋅(sp′−sp)=0 | degree=7

Additionally, we will need to impose a boundary constraint which specifies that the first value in sp=0sp = 0sp=0. Note, however, that we do not need to impose a constraint ensuring that values in spspsp are binary - this will follow naturally from the above constraints.

Also, note that the combination of the above constraints makes it impossible to execute END or RESPAN operations right after SPAN or RESPAN operations.

Block address constraints​

When we are inside a span block, values in block address columns (denoted as aaa) must remain the same. This can be enforced with the following constraint:

sp⋅(a′−a)=0 | degree=2sp \cdot (a' - a) = 0 \text{ | degree} = 2sp⋅(a′−a)=0 | degree=2

Notice that this constraint does not apply when we execute any of the control flow operations. For such operations, the prover sets the value of the aaa column non-deterministically, except for the RESPAN operation. For the RESPAN operation the value in the aaa column is incremented by 888, which is enforced by a constraint described previously.

Notice also that this constraint implies that when the next operation is the END operation, the value in the aaa column must also be copied over to the next row. This is exactly the behavior we want to enforce so that when the END operation is executed, the block address is set to the address of the current span batch.

Group count constraints​

The group_count column (denoted as gcgcgc) is used to keep track of the number of operation groups which remains to be executed in a span block.

In the beginning of a span block (i.e., when SPAN operation is executed), the prover sets the value of gcgcgc non-deterministically. This value is subsequently decremented according to the rules described below. By the time we exit the span block (i.e., when END operation is executed), the value in gcgcgc must be 000.

The rules for decrementing values in the gcgcgc column are as follows:

  • The count cannot be decremented by more than 111 in a single row.
  • When an operation group is fully executed (which happens when h0=0h_0 = 0h0​=0 inside a span block), the count is decremented by 111.
  • When SPAN, RESPAN, or PUSH operations are executed, the count is decremented by 111.

Note that these rules imply that PUSH operation cannot be the last operation in an operation group (otherwise the count would have to be decremented by 222).

To simplify the description of the constraints, we will define the following variable:

Δgc=gc−gc′\Delta gc = gc - gc'Δgc=gc−gc′

Using this variable, we can describe the constraints against the gcgcgc column as follows:

Inside a span block, group count can either stay the same or decrease by one:

sp⋅Δgc⋅(Δgc−1)=0 | degree=3sp \cdot \Delta gc \cdot (\Delta gc - 1) = 0 \text{ | degree} = 3sp⋅Δgc⋅(Δgc−1)=0 | degree=3

When group count is decremented inside a span block, either h0h_0h0​ must be 000 (we consumed all operations in a group) or we must be executing PUSH operation:

sp⋅Δgc⋅(1−fpush)⋅h0=0 | degree=7sp \cdot \Delta gc \cdot (1 - f_{push})\cdot h_0 = 0 \text{ | degree} = 7sp⋅Δgc⋅(1−fpush​)⋅h0​=0 | degree=7

Notice that the above constraint does not preclude fpush=1f_{push} = 1fpush​=1 and h0=0h_0 = 0h0​=0 from being true at the same time. If this happens, op group decoding constraints (described here) will force that the operation following the PUSH operation is a NOOP.

When executing a SPAN, a RESPAN, or a PUSH operation, group count must be decremented by 111:

(fspan+frespan+fpush)⋅(Δgc−1)=0 | degree=7(f_{span} + f_{respan} + f_{push}) \cdot (\Delta gc - 1) = 0 \text{ | degree} = 7(fspan​+frespan​+fpush​)⋅(Δgc−1)=0 | degree=7

If the next operation is either an END or a RESPAN, group count must remain the same:

Δgc⋅(fend′+frespan′)=0 | degree=5\Delta gc \cdot (f_{end}' + f_{respan}') = 0 \text{ | degree} = 5Δgc⋅(fend′​+frespan′​)=0 | degree=5

When an END operation is executed, group count must be 000:

fend⋅gc=0 | degree=5f_{end} \cdot gc = 0 \text{ | degree} = 5fend​⋅gc=0 | degree=5

Op group decoding constraints​

Inside a span block, register h0h_0h0​ is used to keep track of operations to be executed in the current operation group. The value of this register is set by the prover non-deterministically at the time when the prover executes a SPAN or a RESPAN operation, or when processing of a new operation group within a batch starts. The picture below illustrates this.

In the above:

  • The prover sets the value of h0h_0h0​ non-deterministically at row 000. The value is set to an operation group containing operations op0 through op8.
  • As we start executing the group, at every row we "remove" the least significant operation from the group. This can be done by subtracting opcode of the operation from the group, and then dividing the result by 272^727.
  • By row 999 the group is fully executed. This decrements the group count and set op_index to 000 (constraints against op_index column are described in the next section).
  • At row 101010 we start executing the next group with operations op9 through op11. In this case, the prover populates h0h_0h0​ with the group having its first operation (op9) already removed, and sets the op_bits registers to the value encoding op9.
  • By row 121212 this group is also fully executed.

To simplify the description of the constraints, we define the following variables:

op=∑i=06(bi⋅2i)fsgc=sp⋅sp′⋅(1−Δgc)op = \sum_{i=0}^6 (b_i \cdot 2^i) \\ f_{sgc} = sp \cdot sp' \cdot (1 - \Delta gc)op=i=0∑6​(bi​⋅2i)fsgc​=sp⋅sp′⋅(1−Δgc)

opopop is just an opcode value implied by the values in op_bits registers. fsgcf_{sgc}fsgc​ is a flag which is set to 111 when the group count within a span block does not change. We multiply it by sp′sp'sp′ to make sure the flag is 000 when we are about to end decoding of an operation batch. Note that fsgcf_{sgc}fsgc​ flag is mutually exclusive with fspanf_{span}fspan​, frespanf_{respan}frespan​, and fpushf_{push}fpush​ flags as these three operations decrement the group count.

Using these variables, we can describe operation group decoding constraints as follows:

When a SPAN, a RESPAN, or a PUSH operation is executed or when the group count does not change, the value in h0h_0h0​ should be decremented by the value of the opcode in the next row.

(fspan+frespan+fpush+fsgc)⋅(h0−h0′⋅27−op′)=0 | degree=7(f_{span} + f_{respan} + f_{push} + f_{sgc}) \cdot (h_0 - h_0' \cdot 2^7 - op') = 0 \text{ | degree} = 7(fspan​+frespan​+fpush​+fsgc​)⋅(h0​−h0′​⋅27−op′)=0 | degree=7

Notice that when the group count does change, and we are not executing fspanf_{span}fspan​, frespanf_{respan}frespan​, or fpushf_{push}fpush​ operations, no constraints are placed against h0h_0h0​, and thus, the prover can populate this register non-deterministically.

When we are in a span block and the next operation is END or RESPAN, the current value in h0h_0h0​ column must be 000.

sp⋅(fend′+frespan′)⋅h0=0 | degree=6sp \cdot (f_{end}' + f_{respan}') \cdot h_0 = 0 \text{ | degree} = 6sp⋅(fend′​+frespan′​)⋅h0​=0 | degree=6

Op index constraints​

The op_index column (denoted as oxoxox) tracks index of an operation within its operation group. It is used to ensure that the number of operations executed per group never exceeds 999. The index is zero-based, and thus, the possible set of values for oxoxox is between 000 and 888 (both inclusive).

To simplify the description of the constraints, we will define the following variables:

ng=Δgc−fpushΔox=ox′−oxng = \Delta gc - f_{push} \\ \Delta ox = ox' - oxng=Δgc−fpush​Δox=ox′−ox

The value of ngngng is set to 111 when we are about to start executing a new operation group (i.e., group count is decremented but we did not execute a PUSH operation). Using these variables, we can describe the constraints against the oxoxox column as follows.

When executing SPAN or RESPAN operations the next value of op_index must be set to 000:

(fspan+frespan)⋅ox′=0 | degree=7(f_{span} + f_{respan}) \cdot ox' = 0 \text{ | degree} = 7(fspan​+frespan​)⋅ox′=0 | degree=7

When starting a new operation group inside a span block, the next value of op_index must be set to 000. Note that we multiply by spspsp to exclude the cases when the group count is decremented because of SPAN or RESPAN operations:

sp⋅ng⋅ox′=0 | degree=6sp \cdot ng \cdot ox' = 0 \text{ | degree} = 6sp⋅ng⋅ox′=0 | degree=6

When inside a span block but not starting a new operation group, op_index must be incremented by 111. Note that we multiply by sp′sp'sp′ to exclude the cases when we are about to exit processing of an operation batch (i.e., the next operation is either END or RESPAN):

sp⋅sp′⋅(1−ng)⋅(Δox−1)=0 | degree=7sp \cdot sp' \cdot (1 - ng) \cdot (\Delta ox - 1) = 0 \text{ | degree} = 7sp⋅sp′⋅(1−ng)⋅(Δox−1)=0 | degree=7

Values of op_index must be in the range [0,8][0, 8][0,8].

∏i=08(ox−i)=0 | degree=9\prod_{i=0}^{8}(ox - i) = 0 \text{ | degree} = 9i=0∏8​(ox−i)=0 | degree=9

Op batch flags constraints​

Operation batch flag columns (denoted bc0bc_0bc0​, bc1bc_1bc1​, and bc2bc_2bc2​) are used to specify how many operation groups are present in an operation batch. This is relevant for the last batch in a span block (or the first batch if there is only one batch in a block) as all other batches should be completely full (i.e., contain 8 operation groups).

These columns are used to define the following 4 flags:

  • fg8=bc0f_{g8} = bc_0fg8​=bc0​: there are 8 operation groups in the batch.
  • fg4=(1−bc0)⋅bc1⋅bc2f_{g4} = (1 - bc_0) \cdot bc_1 \cdot bc_2fg4​=(1−bc0​)⋅bc1​⋅bc2​: there are 4 operation groups in the batch.
  • fg2=(1−bc0)⋅(1−bc1)⋅bc2f_{g2} = (1 - bc_0) \cdot (1 - bc_1) \cdot bc_2fg2​=(1−bc0​)⋅(1−bc1​)⋅bc2​: there are 2 operation groups in the batch.
  • fg1=(1−bc0)⋅bc1⋅(1−bc2)f_{g1} = (1 - bc_0) \cdot bc_1 \cdot (1 - bc_2)fg1​=(1−bc0​)⋅bc1​⋅(1−bc2​): there is only 1 operation groups in the batch.

Notice that the degree of fg8f_{g8}fg8​ is 111, while the degree of the remaining flags is 333.

These flags can be set to 111 only when we are executing SPAN or RESPAN operations as this is when the VM starts processing new operation batches. Also, for a given flag we need to ensure that only the specified number of operations groups are present in a batch. This can be done with the following constraints.

All batch flags must be binary:

bci2−bci=0 for i∈0..2 | degree=2bc_i^2 - bc_i = 0 \text{ for } i \in 0..2 \text{ | degree} = 2bci2​−bci​=0 for i∈0..2 | degree=2

When SPAN or RESPAN operations is executed, one of the batch flags must be set to 111.

(fspan+frespan)−(fg1+fg2+fg4+fg8)=0 | degree=6(f_{span} + f_{respan}) - (f_{g1} + f_{g2} + f_{g4} + f_{g8}) = 0 \text{ | degree} = 6(fspan​+frespan​)−(fg1​+fg2​+fg4​+fg8​)=0 | degree=6

When we have at most 4 groups in a batch, registers h4,...,h7h_4, ..., h_7h4​,...,h7​ should be set to 000's.

(fg1+fg2+fg4)⋅hi=0 for i∈4..7 | degree=4(f_{g1} + f_{g2} + f_{g4}) \cdot h_i = 0 \text{ for } i \in 4..7 \text{ | degree} = 4(fg1​+fg2​+fg4​)⋅hi​=0 for i∈4..7 | degree=4

When we have at most 2 groups in a batch, registers h2h_2h2​ and h3h_3h3​ should also be set to 000's.

(fg1+fg2)⋅hi=0 for i∈2,3 | degree=4(f_{g1} + f_{g2}) \cdot h_i = 0 \text{ for } i \in 2, 3 \text{ | degree} = 4(fg1​+fg2​)⋅hi​=0 for i∈2,3 | degree=4

When we have at most 1 groups in a batch, register h1h_1h1​ should also be set to 000.

fg1⋅h1=0 | degree=4f_{g1} \cdot h_1 = 0 \text{ | degree} = 4fg1​⋅h1​=0 | degree=4

Op group table constraints​

Op group table is used to ensure that all operation groups in a given batch are consumed before a new batch is started (i.e., via a RESPAN operation) or the execution of a span block is complete (i.e., via an END operation). The op group table is updated according to the following rules:

  • When a new operation batch is started, we add groups from this batch to the table. To add a group to the table, we multiply the value in column p3p_3p3​ by a value representing a tuple (batch_id, group_pos, group). A constraint to enforce this would look as p3′=p3⋅vp_3' = p_3 \cdot vp3′​=p3​⋅v, where vvv is the value representing the row to be added. Depending on the batch, we may need to add multiple groups to the table (i.e., p3′=p3⋅v1⋅v2⋅v3...p_3' = p_3 \cdot v_1 \cdot v_2 \cdot v_3 ...p3′​=p3​⋅v1​⋅v2​⋅v3​...). Flags fg1f_{g1}fg1​, fg2f_{g2}fg2​, fg4f_{g4}fg4​, and fg8f_{g8}fg8​ are used to define how many groups to add.
  • When a new operation group starts executing or when an immediate value is consumed, we remove the corresponding group from the table. To do this, we divide the value in column p3p_3p3​ by a value representing a tuple (batch_id, group_pos, group). A constraint to enforce this would look as p3′⋅u=p3p_3' \cdot u = p_3p3′​⋅u=p3​, where uuu is the value representing the row to be removed.

To simplify constraint descriptions, we first define variables representing the rows to be added to and removed from the op group table.

When a SPAN or a RESPAN operation is executed, we compute the values of the rows to be added to the op group table as follows:

vi=α0+α1⋅a′+α2⋅(gc−i)+α3⋅hi | degree=1v_i = \alpha_0 + \alpha_1 \cdot a' + \alpha_2 \cdot (gc - i) + \alpha_3 \cdot h_{i} \text{ | degree} = 1vi​=α0​+α1​⋅a′+α2​⋅(gc−i)+α3​⋅hi​ | degree=1

Where i∈1..7i \in 1..7i∈1..7. Thus, v1v_1v1​ defines row value for group in h1h_1h1​, v2v_2v2​ defines row value for group h2h_2h2​ etc. Note that batch address column comes from the next row of the block address column (a′a'a′).

We compute the value of the row to be removed from the op group table as follows:

u=α0+α1⋅a+α2⋅gc+α3⋅((h0′⋅27+op′)⋅(1−fpush)+s0′⋅fpush) | degree=5u = \alpha_0 + \alpha_1 \cdot a + \alpha^2 \cdot gc + \alpha_3 \cdot ((h_0' \cdot 2^7 + op') \cdot (1 - f_{push}) + s_0' \cdot f_{push}) \text{ | degree} = 5u=α0​+α1​⋅a+α2⋅gc+α3​⋅((h0′​⋅27+op′)⋅(1−fpush​)+s0′​⋅fpush​) | degree=5

In the above, the value of the group is computed as (h0′⋅27+op′)⋅(1−fpush)+s0′⋅fpush(h_0' \cdot 2^7 + op') \cdot (1 - f_{push}) + s_0' \cdot f_{push}(h0′​⋅27+op′)⋅(1−fpush​)+s0′​⋅fpush​. This basically says that when we execute a PUSH operation we need to remove the immediate value from the table. This value is at the top of the stack (column s0s_0s0​) in the next row. However, when we are not executing a PUSH operation, the value to be removed is an op group value which is a combination of values in h0h_0h0​ and op_bits columns (also in the next row). Note also that value for batch address comes from the current value in the block address column (aaa), and the group position comes from the current value of the group count column (gcgcgc).

We also define a flag which is set to 111 when a group needs to be removed from the op group table.

fdg=sp⋅Δgcf_{dg} = sp \cdot \Delta gcfdg​=sp⋅Δgc

The above says that we remove groups from the op group table whenever group count is decremented. We multiply by spspsp to exclude the cases when the group count is decremented due to SPAN or RESPAN operations.

Using the above variables together with flags fg2f_{g2}fg2​, fg4f_{g4}fg4​, fg8f_{g8}fg8​ defined in the previous section, we describe the constraint for updating op group table as follows (note that we do not use fg1f_{g1}fg1​ flag as when a batch consists of a single group, nothing is added to the op group table):

p3′⋅(fdg⋅u+1−fdg)=p3⋅(fg2⋅v1+fg4⋅∏i=13vi+fg8⋅∏i=17vi−1+(fspan+frespan))p_3' \cdot (f_{dg} \cdot u + 1 - f_{dg}) = p_3 \cdot (f_{g2} \cdot v_1 + f_{g4} \cdot \prod_{i=1}^3 v_i + f_{g8} \cdot \prod_{i=1}^7 v_i - 1 + (f_{span} + f_{respan}))p3′​⋅(fdg​⋅u+1−fdg​)=p3​⋅(fg2​⋅v1​+fg4​⋅i=1∏3​vi​+fg8​⋅i=1∏7​vi​−1+(fspan​+frespan​))

The above constraint specifies that:

  • When SPAN or RESPAN operations are executed, we add between 111 and 777 groups to the op group table.
  • When group count is decremented inside a span block, we remove a group from the op group table.

The degree of this constraint is 999.

In addition to the above transition constraint, we also need to impose boundary constraints against the p3p_3p3​ column to make sure the first and the last value in the column is set to 111. This enforces that the op group table table starts and ends in an empty state.
